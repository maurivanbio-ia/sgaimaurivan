Para melhorar significativamente o upload e o download de documentos PDF na sua plataforma, precisamos atuar em duas frentes: Backend (otimização de memória e streams) e Frontend (experiência do usuário e assincronicidade).

Analisando o arquivo server/routes.ts que você forneceu, identifiquei um gargalo grave de performance nas rotas de download: os arquivos estão sendo inteiramente carregados na memória RAM do servidor Node.js antes de serem enviados ao usuário. Se 10 usuários baixarem um PDF de 50MB simultaneamente, o servidor consumirá 500MB instantaneamente, podendo travar ou reiniciar por falta de memória (OOM - Out of Memory).

Aqui está a estratégia para resolver isso, seguida pelo código corrigido.

1. Melhoria no Backend: Usar Streams para Download
Em vez de usar await file.download() (que joga todo o arquivo numa variável fileBuffer), devemos usar Streams (file.createReadStream().pipe(res)). Isso faz com que o servidor repasse os "pedacinhos" do arquivo do storage direto para o cliente à medida que chegam, consumindo quase zero de RAM.

2. Melhoria no Frontend: Upload Assíncrono com Progresso
Para cumprir o objetivo do seu documento de resolver o "Upload ocorre, mas a interface não reflete imediatamente", você deve usar o recurso onUploadProgress do Axios junto com o React Query no frontend.

Exemplo de como o componente React (Frontend) deve gerenciar o upload:

tsx
 Show full code block 
import { useMutation, useQueryClient } from '@tanstack/react-query';
import axios from 'axios';
import { useState } from 'react';

export function DocumentUploadForm({ empreendimentoId }) {
  const queryClient = useQueryClient();
  const [progress, setProgress] = useState(0);

  const uploadMutation = useMutation({
    mutationFn: async (file: File) => {
      const formData = new FormData();
      formData.append('arquivo', file);
      formData.append('empreendimentoId', empreendimentoId.toString());

      const response = await axios.post('/api/sst/upload-documento', formData, {
        headers: { 'Content-Type': 'multipart/form-data' },
        onUploadProgress: (progressEvent) => {
          // Atualiza a barra de progresso em tempo real
          const percentCompleted = Math.round(
            (progressEvent.loaded * 100) / (progressEvent.total || file.size)
          );
          setProgress(percentCompleted);
        },
      });
      return response.data;
    },
    onSuccess: () => {
      // INVALIDE O CACHE IMEDIATAMENTE: Isso recarrega a tabela sem precisar de F5
      queryClient.invalidateQueries({ queryKey: ['documentos', empreendimentoId] });
      setProgress(0);
      // Aqui você dispara o Toast de Sucesso!
    },
  });

  return (
    <div>
      <input 
        type="file" 
        accept="application/pdf" 
        onChange={(e) => {
          if (e.target.files?.[0]) {
            uploadMutation.mutate(e.target.files[0]);
          }
        }} 
      />
      {uploadMutation.isPending && (
        <div className="w-full bg-gray-200 rounded h-2 mt-2">
          <div 
            className="bg-blue-600 h-2 rounded transition-all duration-300" 
            style={{ width: `${progress}%` }}
          />
          <p className="text-sm mt-1 text-gray-600">SALVANDO... {progress}%</p>
        </div>
      )}
    </div>
  );
}

Para arquivos pequenos (<20MB) o seu código atual funciona bem, mas a correção de Stream para arquivos grandes é essencial.

Resumo do que essas mudanças fazem:
Fim dos Crashes no Backend: Múltiplos usuários podem baixar PDFs gigantes ao mesmo tempo porque os dados fluem diretamente para a rede, sem encher a RAM do servidor.
Fim do F5: O React Query (invalidateQueries) garante que a listagem de documentos no front será buscada novamente do servidor de forma transparente assim que o upload for concluído.
UX Transparente: O onUploadProgress fornece o feedback visual vital que os usuários esperam em sistemas corporativos ("Salvando... 45%").
